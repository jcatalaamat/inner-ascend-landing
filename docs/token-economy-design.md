# Inner Ascend Token Economy Design

## Vision
A biased social media token economy where universal basic income meets contribution rewards in a closed-loop healing ecosystem.

---

## Core Principles

1. **Universal Basic Income (UBI)** - Earn tokens for being present in the community
2. **Contribution Rewards** - Earn more tokens by serving, creating, facilitating
3. **Internal Economy** - Spend tokens within the app (no external extraction)
4. **Community Governance** - Token holders vote on platform direction
5. **Priority Access** - Tokens grant visibility, retreat access, and other benefits

## Token Supply & Sustainability

### Initial Supply
- **10,000,000 $INNER** minted at launch
- Distribution:
  - 40% (4M) - Community rewards pool (UBI + contributions)
  - 30% (3M) - Founding team (vested over 4 years)
  - 20% (2M) - Early members (airdrop for first 1,000 members)
  - 10% (1M) - Reserve for partnerships, grants, emergencies

### Inflation / Deflation
**Inflationary (New tokens minted):**
- 5% annual inflation to fund ongoing UBI
- New tokens minted monthly and added to rewards pool

**Deflationary (Tokens burned):**
- 10% of tokens spent on services are burned (removed from circulation)
- Burned tokens reduce supply, increase scarcity

**Net effect:** Moderate inflation (3-4% annually) to ensure UBI sustainability

### Community Pool Management
- Governed by token holders
- Monthly budget voted on by community
- Transparent on-chain accounting

## Governance System

### Decision Types

#### 1. Small Decisions (App Features)
- Example: "Should we add dark mode?"
- Vote weight: 1 token = 1 vote
- Quorum: 10% of token holders
- Passes with simple majority (>50%)

#### 2. Medium Decisions (Platform Direction)
- Example: "Should we launch a physical retreat center?"
- Vote weight: 1 token = 1 vote, but reputation multiplier applies
- Quorum: 25% of token holders
- Passes with 60% majority

#### 3. Large Decisions (Economic Changes)
- Example: "Should we change UBI rate?"
- Vote weight: Quadratic voting (sqrt of tokens held)
- Quorum: 50% of token holders
- Passes with 75% majority

### Quadratic Voting
**Why?** Prevents whales from dominating governance
- If you hold 100 tokens, your vote weight = sqrt(100) = 10
- If you hold 10,000 tokens, your vote weight = sqrt(10,000) = 100
- Rich members have more influence, but not proportionally more

### Proposal Process
1. **Submit proposal** (costs 100 $INNER, refunded if approved)
2. **Discussion period** (7 days)
3. **Voting period** (7 days)
4. **Execution** (if passed)

## Technical Implementation

### Option 1: Internal Token (Recommended for MVP)
**Database-based, not blockchain:**

**Pros:**
- Simple to build
- No crypto stigma
- Full control over economy
- Fast transactions
- No gas fees

**Cons:**
- Centralized (you control it)
- Could be shut down
- Not "true" web3

**Tech stack:**
- PostgreSQL database for token balances
- Redis for transaction queue
- Node.js backend for token logic
- React frontend for wallet UI

### Option 2: Blockchain Token (Future)
**Real crypto on Ethereum, Polygon, or Solana:**

**Pros:**
- Decentralized (can't be shut down)
- Permissionless (anyone can build on it)
- True community ownership
- Composable with other web3 apps

**Cons:**
- Regulatory risk (SEC, securities law)
- Technical complexity
- Crypto stigma (some people hate crypto)
- Gas fees (expensive on Ethereum)

**Tech stack:**
- Solidity smart contracts
- Hardhat for development
- OpenZeppelin for security
- Web3.js or Ethers.js for frontend
- MetaMask or WalletConnect for user wallets

### Recommended Path
1. **Phase 1 (0-1,000 members):** Internal token, database-based
2. **Phase 2 (1,000-10,000 members):** Hybrid (internal token + governance NFT)
3. **Phase 3 (10,000+ members):** Full blockchain migration

## Legal & Regulatory

### Is This a Security?
**Probably not, if:**
- Tokens are used for access, not investment
- No promise of profit
- Community-governed, not centrally controlled
- Utility token, not equity token

**To be safe:**
- Consult crypto lawyer
- Don't call it an "investment"
- Emphasize utility (access, governance)
- Avoid secondary markets

### Tax Implications
**For members:**
- Earned tokens = income (probably)
- Spent tokens = consumption (probably)
- Need to track cost basis

**For platform:**
- You're facilitating transactions, not selling securities (probably)
- Consult tax lawyer
